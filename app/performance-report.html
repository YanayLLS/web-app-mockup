Level of Detail (LOD) Critical Impact
Swap high-poly meshes for simplified versions at distance. The single biggest triangle-count reduction for large scenes.
▼

The Problem

A mesh with 50K triangles viewed from 100m away might cover 20 pixels on screen — but still sends all 50K triangles through the vertex shader. In a 1M-mesh scene, the majority of visible meshes are distant, wasting 80-90% of GPU vertex throughput on invisible detail.

LOD Strategy for 1M Meshes

  • LOD 0 (0-15m): Full resolution — original mesh geometry
  • LOD 1 (15-50m): 50% triangle reduction — edge-collapse simplification
  • LOD 2 (50-150m): 15% of original — aggressive decimation, preserve silhouette
  • LOD 3 (150m+): 5% or billboard/imposter — flat card with baked render

Implementation Approach

Three.js // Runtime mesh simplification using meshoptimizer WASM import { MeshoptSimplifier } from 'meshoptimizer'; function generateLODChain(geometry, levels = [1.0, 0.5, 0.15, 0.05]) { const lod = new THREE.LOD(); for (let i = 0; i < levels.length; i++) { const simplified = simplifyGeometry(geometry, levels[i]); const mesh = new THREE.Mesh(simplified, material); lod.addLevel(mesh, distances[i]); } return lod; } // Integrate with existing batcher — LOD selection before batching function selectLOD(mesh, cameraPos) { const dist = mesh.position.distanceTo(cameraPos); const screenSize = mesh.boundingSphere.radius / dist * projectionScale; if (screenSize > 100) return 0; // Full detail if (screenSize > 30) return 1; // Half detail if (screenSize > 8) return 2; // Low detail return 3; // Billboard }

Hierarchical LOD (HLOD) Critical Impact
Merge clusters of distant small objects into single proxy meshes. Turns 10,000 tiny screws into one combined draw.
▼

The Problem

Even with instancing, 1M meshes means potentially hundreds of thousands of unique geometries. At distance, an entire building section with 5,000 parts could be replaced by a single merged mesh with baked lighting — 5,000 draw calls → 1.

HLOD Cluster Strategy

  • Leaf clusters: Group meshes within octree leaf nodes (already built!)
  • Merge threshold: When cluster screen-size < 50px, switch to merged proxy
  • Proxy generation: Merge all cluster geometries → simplify to 1-5% → bake single material
  • Seamless transition: Cross-fade or dither between individual meshes and proxy

Leveraging the Existing Octree

Pseudocode // The octree already groups meshes spatially — extend it with HLOD proxies class HLODNode { proxyMesh = null; // Merged + simplified geometry proxyThreshold = 50; // Switch to proxy below 50px screen-size update(camera) { const screenSize = this.getScreenSize(camera); if (screenSize < this.proxyThreshold) { // Hide all child meshes, show single proxy this.children.forEach(c => c.visible = false); this.proxyMesh.visible = true; } else { // Show individual meshes (still culled by occlusion system) this.proxyMesh.visible = false; // Let normal frustum+occlusion handle children } } }

Adaptive Resolution Scaling High Impact
Dynamically lower render resolution during camera motion. Invisible to the user, massive fragment savings.
▼

The Problem

During orbit/pan, the human eye cannot perceive full-resolution detail due to motion blur in biological vision. But the GPU still shades every pixel at 1:1 device pixel ratio. On a 4K display, that's 8.3 million fragments per pass — times 6 passes (SSAO, outlines, etc.).

Strategy

  • Idle: Render at full device pixel ratio (1.0×)
  • Moving: Drop to 0.5–0.75× pixel ratio — 50-75% fewer fragments
  • Fast orbit: Drop to 0.35× — barely noticeable during rapid motion
  • Ramp-up: Smoothly interpolate back to 1.0× over 200ms after motion stops
Three.js let targetRatio = window.devicePixelRatio; let currentRatio = targetRatio; const MOTION_RATIO = 0.5; const RAMP_SPEED = 0.08; function updateResolution(isMoving) { targetRatio = isMoving ? window.devicePixelRatio * MOTION_RATIO : window.devicePixelRatio; currentRatio += (targetRatio - currentRatio) * RAMP_SPEED; renderer.setPixelRatio(currentRatio); } // In orbit controls change handler: controls.addEventListener('change', () => { lastMoveTime = performance.now(); });

GPU Texture Compression (KTX2 / Basis) High Impact
Replace PNG/JPG textures with GPU-native compressed formats. 4-6× VRAM reduction, instant GPU upload.
▼

The Problem

A 2048×2048 PNG becomes a 16MB uncompressed RGBA texture in VRAM, regardless of file size. With 1M meshes potentially sharing hundreds of unique textures, VRAM can blow past GPU limits, causing thrashing and stutters.

BVH-Accelerated Raycasting High Impact
Replace O(n) raycast with O(log n) BVH traversal. Makes hover/click instant even with 1M meshes.
▼

The Problem

Three.js default raycasting iterates every mesh and tests every triangle. With 1M meshes at 50 triangles average = 50M triangle tests per raycast. At 60fps with hover detection, that's 3 billion tests/second. Even with frustum pre-filter, this causes multi-frame hitches.

Solution: three-mesh-bvh

  • Per-geometry BVH: Each mesh gets a bounding volume hierarchy over its triangles
  • Scene-level BVH: Top-level acceleration structure over all mesh AABBs
  • Result: Raycast drops from ~50ms to <0.1ms for 1M meshes
  • Bonus: Also accelerates spatial queries (selection box, proximity tests)
three-mesh-bvh import { computeBoundsTree, disposeBoundsTree, acceleratedRaycast } from 'three-mesh-bvh'; // Monkey-patch Three.js — all meshes get BVH raycasting THREE.BufferGeometry.prototype.computeBoundsTree = computeBoundsTree; THREE.BufferGeometry.prototype.disposeBoundsTree = disposeBoundsTree; THREE.Mesh.prototype.raycast = acceleratedRaycast; // Build BVH after model load (can be done in Worker) scene.traverse(obj => { if (obj.isMesh) obj.geometry.computeBoundsTree(); });

Cascaded Shadow Maps (CSM) Medium Impact
Split the shadow frustum into cascades — crisp near shadows, acceptable far shadows, same memory budget.
▼

The Problem

Current setup: single 2048² VSM shadow map covering ±150m (300m total). That's ~7cm per shadow texel at the near plane — shadows look blurry up close. In CAD/industrial scenes, this is very noticeable on small parts like fasteners and connectors.

Web Worker Offloading High Impact
Move geometry hashing, octree build, AABB computation, and mesh simplification off the main thread.
▼

The Problem

Loading a 1M-mesh model triggers heavy CPU work on the main thread: geometry hashing (sampling vertices), octree construction, AABB computation for culling, and material grouping. This blocks the UI for seconds, showing a frozen loading screen instead of progressive feedback.

What to Offload

Task Current Thread Est. Time (1M meshes) Transferable?
Geometry hashing Main 500-2000ms Yes (ArrayBuffers)
Octree construction Main 200-800ms Yes (typed arrays)
AABB computation Main 300-1000ms Yes (Float32Array)
Mesh simplification (LOD) Main 2000-10000ms Yes (via meshoptimizer WASM)
BVH construction Main 2000-5000ms Yes (serializable)

Architecture

Worker Pool // Main thread — transfer geometry buffers to worker const pool = new WorkerPool(navigator.hardwareConcurrency); async function processModel(meshes) { // Transfer vertex data (zero-copy via Transferable) const tasks = meshes.map(mesh => ({ positions: mesh.geometry.attributes.position.array, indices: mesh.geometry.index?.array, })); // Parallel: hash + AABB + LOD generation const results = await pool.runBatch('processGeometry', tasks); // Back on main thread: build octree from pre-computed AABBs const aabbs = results.map(r => r.aabb); const octree = buildOctreeFromAABBs(aabbs); }

Temporal Reprojection / TAA Medium Impact
Reuse pixel data from previous frames. Amortize expensive shading over time instead of recalculating everything.
▼

The Problem

When the camera is stationary or slow-moving, ~90% of pixels are identical to the previous frame. Yet the entire scene is re-rendered from scratch. SSAO, outlines, and FXAA all re-process every pixel.

Techniques

  • TAA (Temporal Anti-Aliasing): Jitter camera sub-pixel each frame, accumulate over 8-16 frames. Replaces FXAA with superior quality + temporal stability.
  • Temporal SSAO: Accumulate AO samples over frames instead of computing all in one pass. Same quality at 1/4 the per-frame cost.
  • Reprojection Caching: When camera barely moves, reproject last frame and only re-shade disoccluded pixels (advanced, requires motion vectors).

Optimizations

  • Half-res SSAO: Render SSAO at 50% resolution, bilateral upsample → 75% cost reduction
  • Conditional Outlines: Only run outline passes when there's an active selection/hover (skip otherwise)
  • Merge Brightness+FXAA: Combine into single shader pass — eliminates one full-screen blit
  • SSAO → SAO: Switch to Scalable Ambient Obscurance — algorithmically faster, similar quality
